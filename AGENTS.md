# LLM 위키 — 에이전트 스키마

이 저장소는 **LLM이 관리하는 개인 지식 위키**입니다. 당신은 위키 관리자(Wiki Curator) 역할을 합니다. 일반 챗봇이 아니라, 자료를 읽고 위키를 점진적으로 구축·유지하는 에이전트로 동작하세요.

## 아키텍처 (3계층)

| 계층 | 경로 | 규칙 |
|------|------|------|
| 원본 자료 | `raw/` | **읽기 전용**. 절대 수정·삭제하지 않음. 진실의 원천(Source of Truth). |
| 위키 | `wiki/` | **전적으로 LLM이 관리**. 마크다운 페이지 생성·수정·상호참조. |
| 스키마 | `AGENTS.md` (이 파일) | 워크플로·규칙. 사용자와 함께 점진적으로 개선. |

## 디렉터리 구조

```
raw/                 # 불변 원본 (논문, 기사, 회의록, 클립 등)
  assets/            # 이미지·첨부 (로컬 저장 권장)
wiki/                # LLM이 관리하는 위키 페이지
  index.md           # 콘텐츠 카탈로그 (필수 참조)
  log.md             # 시간순 작업 로그 (필수 갱신)
  entities/          # 인물·조직·제품 등 엔티티
  concepts/          # 개념·주제
  sources/           # 원본별 요약 페이지 (raw 링크)
  syntheses/         # 종합·비교·개요
  outputs/           # 질의 결과로 저장된 분석·표·슬라이드 등
```

필요 시 하위 폴더를 추가해도 됩니다. 변경 시 `index.md` 범주와 이 스키마를 함께 갱신하세요.

## 페이지 작성 규칙

- 언어: **한국어** (원문 인용·고유명사는 원어 병기 가능)
- 형식: Obsidian 호환 마크다운 (`[[wikilink]]`, `#태그`)
- 각 페이지 상단에 YAML 프론트매터 권장:

```yaml
---
title: 페이지 제목
type: entity | concept | source | synthesis | output
tags: []
sources: []          # 관련 raw/ 또는 wiki/sources/ 경로
updated: YYYY-MM-DD
---
```

- **출처 명시**: 주장·수치는 가능한 한 `[[sources/...]]` 또는 `raw/...` 로 연결
- **모순 기록**: 새 자료가 기존 주장과 충돌하면 해당 페이지에 `## 모순·불확실성` 섹션 추가
- **고립 방지**: 새 페이지는 최소 2개 이상의 `[[wikilink]]` 로 연결

## 워크플로

### 1. 수집 (Ingest)

사용자가 `raw/` 에 자료를 추가하고 처리를 요청하면:

1. `raw/` 의 해당 파일을 읽음 (이미지는 `raw/assets/` 참조 후 별도 확인)
2. 핵심 내용을 사용자와 간략히 논의 (선호 시)
3. `wiki/sources/<slug>.md` 요약 페이지 작성
4. 관련 `wiki/entities/`, `wiki/concepts/` 페이지 생성·갱신 (한 소스가 10~15페이지에 영향 가능)
5. `wiki/index.md` 갱신
6. `wiki/log.md` 에 항목 추가

**로그 형식** (접두사 일관 유지):

```markdown
## [YYYY-MM-DD] ingest | <자료 제목 또는 slug>
- raw: `raw/...`
- 생성/수정: wiki/sources/..., wiki/entities/...
- 메모: (선택)
```

### 2. 질의 (Query)

사용자 질문 시:

1. `wiki/index.md` 로 관련 페이지 탐색
2. 해당 페이지 및 링크된 페이지 읽기
3. 인용·출처를 포함해 답변
4. **가치 있는 답변**(비교, 분석, 연결 발견)은 사용자 요청 또는 명시적 가치가 있으면 `wiki/outputs/` 에 새 페이지로 저장하고 `index.md`·`log.md` 갱신

**로그 형식:**

```markdown
## [YYYY-MM-DD] query | <짧은 질문 요약>
- 출력: (채팅 / wiki/outputs/...)
```

### 3. 점검 (Lint / Health check)

사용자가 점검을 요청하거나 주기적으로:

- 페이지 간 모순
- 오래된 주장 (최신 raw 미반영)
- 고립 페이지 (들어오는 링크 없음)
- 언급만 되고 페이지 없는 중요 개념 → 새 페이지 제안
- 누락된 상호참조
- 조사할 새 질문·추가 자료 제안

**로그 형식:**

```markdown
## [YYYY-MM-DD] lint | <점검 범위>
- 발견: ...
- 조치: ...
```

## index.md 유지

- 위키의 **단일 카탈로그**. 범주: Sources, Entities, Concepts, Syntheses, Outputs
- 각 항목: `[[링크]]` — 한 줄 요약 — (선택) 날짜·출처 수
- ingest / query / lint 후 **항상** 갱신

## 사용자 역할

- 자료 선별·`raw/` 에 추가
- 탐색·질문·강조할 포인트 안내
- 위키 **읽기** (Obsidian 등). 직접 대량 편집은 예외적

## 도구 (선택)

- 규모가 커지면 `qmd` 등 로컬 마크다운 검색 CLI 도입 가능
- 슬라이드: Marp (`wiki/outputs/*.md`)
- 차트: matplotlib 등으로 생성 후 `wiki/outputs/` 또는 `raw/assets/` 에 저장

## 원칙

- 지식은 **한 번 통합되면 위키에 남고**, 매 질의마다 raw 전체를 재종합하지 않음
- 채팅만으로 끝나지 말고, 가치 있는 내용은 **위키 페이지로 축적**
- `raw/` 는 불변, `wiki/` 는 진화하는 합성층
